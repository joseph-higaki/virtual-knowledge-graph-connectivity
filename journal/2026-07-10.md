## Decisions (and why)
- **Queries bind by label, project only literals.** Ontop mints `het.io/…` IRIs, GraphDB uses
  `ncbigene:`/`do:`; a label-bound, label-projected query runs unchanged on both, and parity never
  compares IRIs (CLAUDE.md #7).
- **`ingest/` and `ui/` are dev packages, not shipped.** Not in `pyproject` `packages`; run via
  `python -m` from repo root. Keeps the installable surface to `harness`.
- **Label projection = drop URI-typed columns.** General for arbitrary UI queries; required `fetch()`
  to preserve binding `type`, so `run_query` was refactored rather than duplicated.
- **Compare UI is a local live app, not a Claude Artifact.** Artifacts run under a CSP that blocks all
  network calls, so they can't query live endpoints. A stdlib server proxying server-side also
  sidesteps CORS and keeps comparison logic in tested Python.
- **`parity.py` stays pure SPARQL.** Source-count attribution is an optional param, not a Postgres
  coupling.

  changed inode — must **recreate** (`make down && make up-rung2`). Will recur every rung that touches
  `.obda`/properties.
- **UI timing can mislead:** the first query after an Ontop (re)start pays cold-connection/JIT cost, so
  the virtual latency bar looks high (~195 ms) then drops to ~15–20 ms on re-run. Honest data, but note
  it in any demo.
- **Not an apples-to-apples benchmark:** GraphDB holds the full Hetionet (~2.25M edges); parity holds
  because the slice is *complete for the queried predicates*. The comparison proves answer parity, not
  performance parity.
