## Decisions (and why)
- **Nessie catalog, in-memory store.** Lightest that works locally + mirrors the Ontop team's
  known-working reference (de-risks the Ontop path); re-checked vs Trino 478 docs. In-memory means
  catalog pointers are ephemeral — reload via `make load-iceberg`, which matches every rung's
  `up → load` flow. Data persists in MinIO.
- **Second Ontop service on the same host port `:7300`.** Keeps the "system under test" endpoint
  constant across rungs so the harness config never changes; only one rung's profile runs at a time.
- **Trino holds S3 creds via `${ENV:…}`**, not the tracked properties file — same secret-injection
  pattern as Ontop's `ONTOP_DB_*`.
- **Fully-qualified `iceberg.hetionet.compound` in the mapping**, but the JDBC URL's default
  catalog/schema lets Ontop emit bare `"compound"` to Trino — satisfies CLAUDE.md's rung-3 addressing
  and pre-figures rung 4.
